# Module 17: AI and ML for Product Managers

## Інструменти

- **Визначення та основи analytics** — описова аналітика (що сталося), діагностична (чому), предиктивна (що станеться), прескриптивна (що робити); рівні складності та вимоги до даних.
- **Визначення та основи AI/ML** — ML як підмножина AI; навчання на даних, моделі (класифікація, регресія, рекомендації, NLP, комп’ютерний зір); supervised vs unsupervised; які задачі продукту можна вирішувати ML.
- **AI/ML платформи та інструменти** — хмарні сервіси (AWS SageMaker, GCP Vertex, Azure ML), no-code/low-code (DataRobot, etc.), вбудовані API (рекомендації, переклад, тощо); коли будувати власне, коли використовувати готове.
- **Типи аналітики для продукту** — descriptive (дашборди, звіти), predictive (прогнози churn, LTV, потреб), prescriptive (рекомендації дій, автоматизація рішень); зв’язок з фічами продукту.
- **Роль PM в AI/ML** — визначення use cases, вимоги до якості та fairness, прийняття ризиків (помилки моделі, explainability); не потрібно писати моделі, але потрібно розуміти можливості та обмеження.

## Висновки (запам’ятати та використовувати)

1. **AI/ML — інструмент для продуктових задач, а не мета сама по собі:** спочатку проблема користувача та бізнес-результат, потім питання “чи ML тут допоможе?”; інакше ризик “AI для галочки”.
2. **Якість даних визначає якість ML:** поганий або зміщений датасет дає несправедливу або ненадійну модель; PM має враховувати джерела даних і обмеження при постановці вимог.
3. **Descriptive → Predictive → Prescriptive — зростаюча складність:** починай з опису (дашборди, події), потім прогнози, потім рекомендації/автоматизація; не стрибай у складне без основ.
4. **Explainability та fairness — частина продуктових вимог:** де рішення впливає на людей, потрібна зрозумілість і перевірка на bias; включай це у вимоги до ML-фіч.
5. **PM не будує моделі, але формулює критерії успіху:** що означає “добре” для рекомендації, прогнозу чи класифікації; як вимірювати і як обмежувати шкоду при помилках моделі.
